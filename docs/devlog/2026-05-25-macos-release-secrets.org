* [2026-05-25] Dev Log: macOS release signing secrets       :RELEASE:SECURITY:OPS:

** Context
PingBar now has a tag-triggered release path:

- =.github/workflows/release.yml= imports a Developer ID certificate into a temporary GitHub Actions keychain.
- =scripts/package-dmg.sh= archives the app, exports a Developer ID signed build, creates and signs a DMG, notarizes it, staples it, and writes a SHA-256 checksum.
- =docs/releasing.md= describes the release command and the public artifact checks.

The remaining setup is not code. It is account and secret hygiene around Apple Developer signing.

** Frame
There are three separate trust boundaries:

| Boundary | Credential | Purpose |
|----------+------------+---------|
| Code signing | Developer ID Application certificate plus private key | Proves the app came from our Apple Developer team. |
| Notarization | Apple ID app-specific password, or a local =notarytool= keychain profile | Lets Apple scan and notarize the signed DMG. |
| GitHub Actions | Repository secrets | Moves the signing/notary credentials into a short-lived runner keychain. |

Do not collapse these together. A =.p12= cannot notarize, an app-specific password cannot sign, and the GitHub secret store is only the transport layer.

** Required GitHub secrets
| Secret | Meaning | Source |
|--------+---------+--------|
| =APPLE_TEAM_ID= | Apple Developer Team ID, usually 10 characters. | Apple Developer account, or the suffix in the signing identity. |
| =APPLE_ID= | Apple ID email used for notarization. | Apple account with access to the developer team. |
| =APPLE_APP_SPECIFIC_PASSWORD= | App-specific password for notarization. | appleid.apple.com account security page. |
| =DEVELOPER_ID_APPLICATION= | Full signing identity string. | =security find-identity -v -p codesigning= on macOS. |
| =MACOS_CERTIFICATE_P12= | Base64-encoded =.p12= containing Developer ID cert and private key. | Keychain Access export. |
| =MACOS_CERTIFICATE_PASSWORD= | Password chosen when exporting the =.p12=. | Human-created export password. |

The current GitHub workflow expects Apple ID notarization secrets. The packaging script also supports =NOTARYTOOL_PROFILE= for local releases, but that profile is a macOS keychain item and is not portable to GitHub Actions.

** macOS host setup
These commands must run on the macOS host, not inside the Linux agent container.

*** Create or confirm the signing certificate
1. In Apple Developer, create or confirm a *Developer ID Application* certificate for the team.
2. Install it in Keychain Access with the private key.
3. Confirm the identity is usable:

#+BEGIN_SRC bash
security find-identity -v -p codesigning
#+END_SRC

The identity should look like:

#+BEGIN_SRC text
Developer ID Application: Example LLC (TEAMID)
#+END_SRC

Use that full string as =DEVELOPER_ID_APPLICATION=.

*** Export the certificate for CI
In Keychain Access:

1. Select the Developer ID Application certificate and its private key.
2. Export as =DeveloperIDApplication.p12=.
3. Use a strong one-time export password.
4. Store that export password as =MACOS_CERTIFICATE_PASSWORD=.

Encode the =.p12= without line breaks:

#+BEGIN_SRC bash
base64 -i DeveloperIDApplication.p12 | tr -d '\n' | pbcopy
#+END_SRC

Paste the clipboard value into =MACOS_CERTIFICATE_P12=. Delete the local =.p12= when setup is complete, or move it to an encrypted secrets store. Do not commit it.

*** Create notarization credentials
For GitHub Actions, create an Apple app-specific password and store:

#+BEGIN_SRC text
APPLE_ID=apple-id@example.com
APPLE_APP_SPECIFIC_PASSWORD=xxxx-xxxx-xxxx-xxxx
APPLE_TEAM_ID=TEAMID
#+END_SRC

For local releases, prefer a stored =notarytool= profile so the app-specific password is not repeated in shell history:

#+BEGIN_SRC bash
xcrun notarytool store-credentials pingbar-notary \
  --apple-id "$APPLE_ID" \
  --team-id "$APPLE_TEAM_ID" \
  --password "$APPLE_APP_SPECIFIC_PASSWORD"
#+END_SRC

Then local packaging can use:

#+BEGIN_SRC bash
DEVELOPER_ID_APPLICATION="Developer ID Application: Example LLC (TEAMID)" \
APPLE_TEAM_ID="TEAMID" \
NOTARYTOOL_PROFILE=pingbar-notary \
make release-dmg VERSION=0.2.0
#+END_SRC

** Upload to GitHub
Use =gh secret set= from the PingBar repo. Avoid putting sensitive values directly in shell history with =--body=.

#+BEGIN_SRC bash
cd /Users/eric/wrk/speedtest/worktree/pingbar

printf '%s' "$APPLE_TEAM_ID" | gh secret set APPLE_TEAM_ID
printf '%s' "$APPLE_ID" | gh secret set APPLE_ID
printf '%s' "$APPLE_APP_SPECIFIC_PASSWORD" | gh secret set APPLE_APP_SPECIFIC_PASSWORD
printf '%s' "$DEVELOPER_ID_APPLICATION" | gh secret set DEVELOPER_ID_APPLICATION
printf '%s' "$MACOS_CERTIFICATE_P12" | gh secret set MACOS_CERTIFICATE_P12
printf '%s' "$MACOS_CERTIFICATE_PASSWORD" | gh secret set MACOS_CERTIFICATE_PASSWORD

gh secret list
#+END_SRC

The variables can come from a password manager, a private shell session, or interactive prompts. The important rule is that real values never land in the repo, PR comments, terminal transcripts shared publicly, or CI logs.

** Validation
Before tagging a public release, run a local signing smoke test on macOS:

#+BEGIN_SRC bash
DEVELOPER_ID_APPLICATION="Developer ID Application: Example LLC (TEAMID)" \
APPLE_TEAM_ID="TEAMID" \
SKIP_NOTARIZATION=1 \
make release-dmg VERSION=0.2.0
#+END_SRC

This proves archiving, export signing, DMG creation, and DMG signing. Do not publish this DMG.

For a full local release rehearsal:

#+BEGIN_SRC bash
DEVELOPER_ID_APPLICATION="Developer ID Application: Example LLC (TEAMID)" \
APPLE_TEAM_ID="TEAMID" \
NOTARYTOOL_PROFILE=pingbar-notary \
make release-dmg VERSION=0.2.0
#+END_SRC

The script should verify codesign, notarize, staple, validate Gatekeeper acceptance, and write:

#+BEGIN_SRC text
build/releases/PingBar-0.2.0.dmg
build/releases/PingBar-0.2.0.dmg.sha256
#+END_SRC

** Release trigger
After the PR is merged and secrets are present, a tag starts the release:

#+BEGIN_SRC bash
git tag v0.2.0
git push origin v0.2.0
#+END_SRC

The GitHub release workflow derives =VERSION= from the tag name. If =v0.2.0= does not match the app's =CFBundleShortVersionString=, packaging should fail instead of publishing a mislabeled build.

** Failure modes
| Symptom | Likely cause | Fix |
|---------+--------------+-----|
| GitHub says =Missing secret ...=. | Secret absent or named differently. | Recheck =gh secret list= and workflow secret names. |
| =security find-identity= in CI shows no usable identity. | =.p12= did not include the private key, or password is wrong. | Re-export cert plus private key from Keychain Access. |
| =xcodebuild -exportArchive= cannot find signing identity. | =DEVELOPER_ID_APPLICATION= does not exactly match the imported identity. | Copy the full identity from =security find-identity=. |
| Notarization fails with authentication/team errors. | Wrong Apple ID, app-specific password, or Team ID. | Regenerate app-specific password and confirm team membership. |
| Packaging refuses =get-task-allow=. | Debug entitlement leaked into release export. | Keep release entitlements free of =com.apple.security.get-task-allow=. |
| Version mismatch failure. | Tag and =project.yml= marketing version differ. | Bump project version before tagging. |

** Rotation
If the app-specific password leaks, revoke it immediately at appleid.apple.com and replace =APPLE_APP_SPECIFIC_PASSWORD=.

If the =.p12= or its password leaks, assume the Developer ID private key is compromised. Revoke the certificate in Apple Developer, create a new Developer ID Application certificate, export a new =.p12=, and replace =MACOS_CERTIFICATE_P12=, =MACOS_CERTIFICATE_PASSWORD=, and possibly =DEVELOPER_ID_APPLICATION=.

** Decision
Keep GitHub release automation narrow and explicit: one tag, one signed/notarized DMG, one checksum. The release workflow should fail closed on missing secrets, signing mismatch, notarization failure, debug entitlements, or version drift.
